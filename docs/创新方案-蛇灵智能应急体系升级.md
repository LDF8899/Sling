# 蛇灵 (SLING) — 智能应急体系升级方案

> "从一个蛇类识别工具，升级为 AI 驱动的蛇伤应急智能体"

---

## 一、叙事主线

### 1.1 现状：工具阶段

蛇灵目前是一个**信息查询平台**：用户拍照识别蛇种 → 查看急救信息 → 搜索附近医院 → 一键拨 120。

这套链路能用，但有一个根本问题——**每个环节都需要人来决策**。用户自己判断严不严重，用户自己决定去哪家医院，救助员自己手动派单，管理员自己盯着血清库存。

### 1.2 目标：智能体阶段

升级后的蛇灵是一个**有自主决策能力的应急智能体**。它不只是告诉你"这是银环蛇，有毒"，而是：

```
拍照 → AI 确认蛇种 → 自动判断严重等级 → 自动找到有血清的最近医院
    → 自动通知医院准备血清 → 自动派遣最近的救助员 → 自动追踪直到脱险
    → 自动复盘存入病例库 → 反哺下一次决策
```

人从"操作者"变成"监督者"——AI 先做决策，人来确认或纠正。

### 1.3 核心技术叙事：中间件即能力

| 中间件 | 在叙事中的角色 | 一句话定义 |
|--------|---------------|-----------|
| **RabbitMQ** | 神经系统 | 事件驱动，让各服务实时协同 |
| **Neo4j** | 大脑皮层 | 关系推理，血清-蛇种-医院的知识图谱 |
| **Chroma** | 长期记忆 | 病例向量化，相似案例检索 |
| **Redis** | 工作记忆 | GEO 空间索引 + 实时状态缓存 |
| **MongoDB** | 日志系统 | 事件溯源 + 行为分析 |
| **MinIO** | 资产仓库 | 图片/离线包/模型文件存储 |
| **SearXNG** | 外部感知 | 联网搜索增强 LLM 回答 |
| **WebSocket** | 实时触觉 | 指挥大屏的实时数据推送 |

## 二、三大端的升级蓝图

### 2.1 用户端（小程序）—— 从"查询工具"到"随身安全助手"

#### 现状
拍照 → 看结果 → 自己查医院 → 自己打电话

#### 升级后

**场景 A：正常识别**
```
用户拍照 → AI 识别出竹叶青（置信度 94%）
         → 系统自动推送：
           "当前位置 3km 内有 2 家医院有竹叶青血清
            → XX医院（2.1km，库存 12 支，预计 8 分钟到达）
            → YY医院（2.8km，库存 3 支，预计 11 分钟到达）
            推荐 XX医院，已为您生成导航路线"
```

**场景 B：紧急 SOS（被咬了）**
```
用户点 SOS → 自动上传位置 + 照片
           → 后台 AI Agent 启动全链路：
             ① 识别蛇种（从照片或用户补充）
             ② 判断严重等级（神经毒素=最高级）
             ③ Neo4j 查询所有有效血清
             ④ Redis GEO 找最近有库存的医院
             ⑤ RabbitMQ 推送：
                → 医院端收到："预计 8 分钟后到达一例银环蛇咬伤患者，请准备抗银环蛇毒血清"
                → 最近的救助员收到："距离 1.2km 有 SOS，请前往协助"
             ⑥ WebSocket 推送给用户：
                "已通知 XX 医院，救助员张三正在赶来（预计 5 分钟），请保持不动"
```

**场景 C：离线模式（深山无网）**
```
无网 → 离线识别（本地 TensorFlow.js 轻量模型）
    → 离线急救手册（本地 JSON，204 种蛇的急救信息）
    → 离线 SOS 缓存（GPS + 时间 + 照照，本地存储）
    → 恢复网络后自动上传，触发完整链路
```

### 2.3 管理端（Admin）—— 从"数据表格"到"数据智能"

#### 现状
看表格 → 手动增删改查 → 手动分析

#### 升级后

**A. 数字孪生沙盘**（Neo4j 驱动）

管理端新增一个"全局态势"页面，用 Cytoscape.js 渲染整个应急体系的关系图：

```
节点类型：
  🐍 蛇种（204 个）—— 按科/属分组
  🏥 医院（1500+）—— 按区域分组
  💉 血清库存 —— 大小映射库存量
  ⚠️ 预警区域 —— 颜色映射风险等级
  📍 历史案例 —— 按时间聚合

关系类型：
  蛇种 --[produces]--> 毒素
  毒素 <--[neutralizes]-- 血清
  血清 --[stocked_at]--> 医院
  医院 --[covers]--> 区域
  蛇种 --[active_in]--> 区域
  案例 --[involved]--> 蛇种 + 医院 + 救助员
```

交互方式：
- 点击一个**区域** → 展开该区域所有蛇种 + 对应血清覆盖情况
- 点击一个**蛇种** → 展开所有有该血清的医院 + 库存量
- 点击一家**医院** → 展开其血清覆盖的蛇种范围 + 未覆盖的高危蛇种
- 搜索"银环蛇" → 高亮所有相关节点：分布区域、有血清的医院、历史案例

**B. 血清供应链智能**

```
血清管理 → 不再是手动录入表格，而是：
  ① 预测消耗：根据历史 SOS + 预警数据，预测未来 7 天各区域血清需求
  ② 库存预警：低于安全阈值自动标红 + 推送通知
  ③ 过期预警：临期血清优先推荐使用
  ④ 调拨建议：A 医院库存紧张 + B 医院库存充足 + 距离近 → 自动生成调拨单
  ⑤ 采购建议：某类血清全区域紧张 → 生成采购报告
```

**C. 病例智能分析**（Chroma RAG 驱动）

## 三、技术架构升级

### 3.1 新增服务：agent-service

```
agent-service（端口 7010）
├── 监听 RabbitMQ 事件队列
├── Agent 编排引擎
│   ├── RecognitionAgent — 蛇种确认 + 严重等级判断
│   ├── DispatchAgent — 医院推荐 + 救助员派遣
│   ├── NotificationAgent — 多端通知（医院/救助员/用户/管理员）
│   ├── TrackingAgent — 状态追踪 + 超时升级
│   └── AnalysisAgent — 病例分析 + 预测
├── WebSocket 推送服务
└── 事件溯源写入 MongoDB
```

### 3.2 RabbitMQ 事件总线

```
Exchange: sling.events (topic)

Queues:
  sos.created          → agent-service (触发 Agent 编排)
  sos.status.changed   → ws-service (推送给指挥中心)
  serum.low_stock      → notification-service (库存预警)
  warning.upgraded     → ws-service (推送给用户端)
  case.completed       → chroma-service (向量化存储)
  case.completed       → mongo-service (事件溯源)

Routing Key 格式:
  sos.{type}.{region}       例: sos.snake_bite.guangdong
  serum.{action}.{hospital}  例: serum.low_stock.hospital_123
  warning.{level}.{region}   例: warning.upgraded.shenzhen_nanshan
```

### 3.3 数据流全景

```
用户拍照 ─→ recognition-service ─→ AI 识别
    │                                    │
    │         ┌──────────────────────────┘
    │         ▼
    │    identify-full 返回：蛇种 + 急救信息 + 医院列表
    │         │
    ▼         ▼
 用户点 SOS ─→ emergency-service ─→ 存入 MySQL
    │              │
    │              ├─→ RabbitMQ: sos.created
    │              │        │
    │              │        ▼
    │              │   agent-service（AI Agent 编排）
    │              │        │
    │              │        ├─→ Neo4j: 查询有效血清
    │              │        ├─→ Redis GEO: 找最近医院
    │              │        ├─→ hospital-service: 确认库存
    │              │        ├─→ RabbitMQ: 通知各端
    │              │        │        │
    │              │        │        ├─→ WebSocket → 指挥中心大屏
    │              │        │        ├─→ WebSocket → 用户端（"正在处理"）
    │              │        │        └─→ 推送 → 医院端 + 救助员端
    │              │        │
    │              │        └─→ MongoDB: 事件溯源
    │              │
    │              └─→ Chroma: 病例向量化存储
    │
    ▼
 事件完成后 ─→ MongoDB 行为流 ─→ 数据分析/报表
```

### 3.4 中间件职责矩阵

| 中间件 | 新职责 | 数据格式 | 关键命令/接口 |
|--------|--------|----------|-------------|
| **RabbitMQ** | 事件总线 | JSON 事件 | topic exchange + 5 个 queue |
| **Neo4j** | 血清关系图谱 | 节点 + 关系 | Cypher 查询最短血清路径 |
| **Chroma** | 病例向量化 | 文本 embedding | 相似度检索 top-K |
| **Redis** | GEO 医院索引 + Agent 状态 | Hash / GEO | GEOSEARCH + HSET |
| **MongoDB** | 事件溯源 + 行为流 | BSON 文档 | 聚合管道分析 |
| **WebSocket** | 实时推送 | JSON 消息 | STOMP over WebSocket |
| **MinIO** | 离线包 + 模型文件 | 二进制 | 预签名 URL 下载 |
| **SearXNG** | 联网搜索增强 | HTML → 结构化 | HTTP API → LLM 上下文 |

## 六、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| AI Agent 决策错误 | 误判严重等级，派错医院 | 人机协同：AI 建议 + 人工确认，不自动执行高风险操作 |
| 服务雪崩 | 一个服务挂了全链路断 | Sentinel 熔断 + RabbitMQ 消息持久化 + 重试机制 |
| 数据一致性 | 多服务状态不同步 | Seata 分布式事务（已有）+ 事件溯源补偿 |
| 离线识别准确率低 | 本地模型不如云端 | 仅作为应急降级方案，有网时自动切回云端 |
| 中间件运维复杂 | 8 个中间件维护成本 | Docker Compose 一键部署 + 健康检查 + 自动重启 |
