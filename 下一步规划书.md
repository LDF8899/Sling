## 四、后端服务规划（P1）

### 4.1 recognition-service — 识别服务改造

这是核心的"救命链路"关键环节。

**当前状态**: 基础框架存在，未对接 Redis

**改造任务**:
- [ ] AI 识别返回蛇名后，通过 Redis `sling:snake:name:{name}` 查 snake_id
- [ ] 用 snake_id 查 `sling:snake:info:{id}` Hash 获取急救卡片
- [ ] 返回格式：{ snake_info, emergency_card: { symptoms, first_aid, forbidden, serum_type, department }, nearby_hospitals: [...] }
- [ ] 同时调用 hospital-service 获取附近医院

**接口设计**:
```
POST /api/recognition/identify
  ↓ 上传图片
  ↓ AI 识别返回蛇名
  ↓ Redis 查急救卡片
  ↓ Feign 调用 hospital-service 查附近医院
  ↓ 组装返回
```

**预计工作量**: 2-3 周

## 九、风险与应对

### 9.1 数据风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 急救信息缺乏可靠来源 | 中 | 高 | 以《中国蛇类》、《中国动物志》等权威著作为主，医学术期刊为辅 |
| 医院血清数据难获取 | 高 | 中 | 优先收集公开名单，电话核实为辅；标注数据来源和不确定性 |
| 蛇类分类争议（新种/隐存种） | 低 | 低 | 采用最新分类学文献，标注争议项 |

### 9.2 技术风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 火山引擎 API 审核不通过 | 中 | 高 | 准备备选方案：百度识图 API / 本地模型（ResNet） |
| AI 识别准确率不达标 | 中 | 中 | 限定识别范围（仅中国蛇类）、多角度+多图片、人工审核兜底 |
| 小程序审核被拒 | 低 | 中 | 提前准备医疗资质说明、避免敏感权限 |
| 本地离线 JSON 体积过大 | 低 | 低 | 仅打包核心急救字段，图片使用 CDN + 缓存 |

### 9.3 时间风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 数据补全耗时超预期 | 高 | 高 | 分阶段：先完成全部毒蛇（~94种）+ 常见无毒蛇，其余逐步补充 |
| 课业与项目冲突 | 高 | 中 | 利用暑假（7-8月）集中开发，学期内维持每周 10h 底线 |
| 前后端联调延迟 | 中 | 中 | 先定义接口文档（API-first），前后端并行开发 |

### 9.4 资源风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| API 调用费用超预算 | 低 | 中 | 设置每日限额，Redis 缓存减少重复调用 |
| 服务器性能不足 | 低 | 低 | 学生优惠机 + Docker 压缩部署 |
